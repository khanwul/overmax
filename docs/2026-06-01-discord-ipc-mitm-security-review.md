# Discord RPC IPC 기반 씬 감지 기술 분석 및 위험성 검토 보고서

본 문서는 오버레이 시스템의 씬 감지 정확도와 성능 향상을 위해 진행된 **Discord IPC Named Pipe 기반 씬 감지 기술**의 실증 연구 내용, 검증 결과 및 보안·안정성 리스크로 인한 최종 검토 의견을 기록합니다.

---

## 1. 연구 배경 및 목적
기존의 OCR 및 픽셀 유사도 분석 기반 씬 감지(L2 Distance, MAD) 방식은 인게임 BGA(백그라운드 애니메이션)의 반투명 투과 및 테마 변경 등 외부 그래픽 노이즈에 매우 취약했습니다.

DJMAX RESPECT V 게임이 디스코드 공식 연동(Discord Rich Presence)을 위해 백그라운드로 송출하는 **Windows Named Pipe IPC 데이터**를 가로채거나 릴레이함으로써, 그래픽 노이즈 없이 CPU 소모율이 거의 0에 수렴하는 극저부하 씬 감지 시스템을 구축하고자 연구를 시작했습니다.

---

## 2. Technical Implementation & Verification Results

### 가. Windows Named Pipe 구조 분석
Windows 환경에서 디스코드 RPC 클라이언트는 다음 경로로 명명된 파이프(Named Pipe)를 생성하여 통신합니다.
* `\\.\pipe\discord-ipc-0` ~ `\\.\pipe\discord-ipc-7`
* 통신 프로토콜 프레임 구조: `[Opcode (u32)][Length (u32)][JSON Payload]`

### 나. 가상 IPC 서버 및 1:N 릴레이 구조 설계
디스코드 클라이언트 기동 여부와 무관하게 데이터를 확보하기 위해 아래와 같은 구조를 구축하고 검증했습니다.
1. **파이프 선점 및 가상 서버**: `discord-ipc-0` 파이프를 먼저 Listen 모드로 열어 둡니다.
2. **윈도우 목업**: 디스코드 SDK가 연결 대상을 탐지할 수 있도록 백그라운드에 보이지 않는 `Chrome_WidgetWin_1` 클래스명과 `Discord` 타이틀을 가진 윈도우 창을 활성화합니다.
3. **RPC 데이터 파이싱**: 게임 프로세스가 가상 파이프에 접속하여 전송하는 JSON 페이로드에서 `activity.details` 문자열을 파싱해 곡명, 키 모드, 난이도 및 씬 상태를 100% 신뢰도로 추출합니다.
4. **브로드캐스팅 릴레이**: 실제 디스코드 클라이언트가 실행될 경우, 수신한 게임 상태 데이터를 사용 중인 다른 파이프 채널(`discord-ipc-1` 등)로 복제 송신(Broadcasting)하여 공식 디스코드 연동 기능이 정상 작동하도록 중계합니다.

## 3. 실증 시나리오 검증
* **가상 서버 선점 후 디스코드 구동 (성공)**: 디스코드가 자동으로 `discord-ipc-1`에 기동되고, 중계 데몬이 이를 인지하여 릴레이를 정상적으로 수행했습니다.
* **중개 중 디스코드 종료/재구동 (성공)**: 파이프 유실 시 릴레이 스트림을 즉시 닫고, 디스코드 재기동 시 자동으로 채널을 재매핑하여 복원했습니다.
* **일회성 초기화 제약 극복**: 게임 내 Discord SDK는 기동 시 단 한 번만 파이프 스캔을 시도하므로, 항상 상시 작동(Daemonized)하는 방식으로 배포해야 연동 안정성이 확보됨을 실증했습니다.

## 4. 리스크 검토 및 보류 판정 (중요)

실제로 시나리오 검증 및 꽤 높은 신뢰도로 파이프라인 검증까지 마쳤으나, 다음과 같은 심각한 잠재적 위험성이 식별되어 메인 파이프라인 적용 및 배포를 **최종 보류**하였습니다.

### 1) 안티치트 솔루션(XIGNCODE3) 탐지 리스크 (MitM 기법의 위험성)
* 본 기술은 게임 클라이언트와 디스코드 프로세스 간의 IPC 통신 경로 중간에 개입하는 **Man-in-the-Middle (MitM) 형태**로 동작합니다.
* 현재는 XIGNCODE3 등의 보안 솔루션에 탐지되지 않더라도, 파이프 선점 및 가상 윈도우 클래스 목업 행위는 보안 솔루션 관점에서 **악성 코드 혹은 불법 프로그램의 은닉/후킹 행위로 오인될 가능성**이 매우 높습니다. 
* 이는 향후 보안 패치에 따라 오버레이 사용자들의 게임 계정 제재 및 실행 차단과 같은 치명적인 부작용을 야기할 수 있습니다.

### 2) 비공식 API 이용에 따른 유지보수 불안정성
* 게임 개발사나 디스코드 측에서 공식적으로 제공하는 외부 노출 API가 아니며, 디스코드 클라이언트 SDK의 내부 동작 방식을 역공학(Reverse Engineering)하여 구현한 비공식(Unofficial) 인터페이스에 의존합니다.
* 디스코드 Rich Presence 스펙이나 윈도우 매핑 룰이 패치로 변경될 경우 전체 탐지 로직이 무력화될 수 있습니다.

---

## 5. 최종 결론
> IPC 기반 씬 감지는 성능 및 정확도 측면에서 완벽한 개선을 보장하지만, **사용자 보안 환경(안티치트 오탐지 등)에서의 안정성을 위협할 가능성**이 존재합니다. 
> 
> 따라서, 안정적인 서비스 유지를 위해 본 기술은 **실험 단계로 격리**하며, 프로덕션 배포 파이프라인에서는 제외하고 기존의 안전한 **화면 캡처 및 이미지 분석 기반 파이프라인을 유지 및 지속 개선**하는 방향을 권장합니다.
