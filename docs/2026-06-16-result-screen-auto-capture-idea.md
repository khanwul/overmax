# 2026-06-16 결과 화면 자동 캡처 기능 기획 및 한계점 정리

이 문서는 플레이어의 신기록(Personal Best) 또는 퍼펙트 플레이 달성 시 결과 화면을 자동으로 캡처하여 저장하는 기능의 기획안과, 현재 바로 구현하기 어려운 기술적 한계점 및 향후 우아한 해결을 위한 아키텍처 설계 방향을 정리한다.

---

## 1. 기능 개요
* **목적**: 플레이어가 곡을 완주한 후 결과 화면(Result Screen)에서 최고 기록을 경신했거나 퍼펙트 플레이(100.00%)를 달성했을 때, 별도의 단축키 조작 없이 당시의 결과 화면을 고화질 이미지로 자동 저장하여 소장용 앨범을 제공한다.
* **트리거 조건**:
  * 감지된 판정 수치(Rate) > 로컬 SQLite DB의 기존 최고 기록 (OR 기존 기록이 없는 최초 플레이)
  * 감지된 판정 수치(Rate) == 100.00% 이며 MAX COMBO 달성 시

---

## 2. 현재 바로 구현할 수 없는 기술적 한계점

현재 Overmax의 아키텍처 상 이 기능을 즉시 도입하기 어려운 구체적인 한계는 다음과 같다.

### 2.1. 결과 화면 씬 감지 기능의 부재 (`SceneType` 미지원)
* **현황**: 현재 `SceneType`은 [game_state.rs](file:///C:/Users/jeongwoong/dev/overmax/rust/overmax_core/src/game_state.rs#L5-L9)에 정의된 것처럼 `Freestyle` 및 `Online` 대기방만 지원한다.
* **한계**: 결과 화면에 진입했는지 여부를 판별할 수 있는 OCR 키워드(예: "RESULT", "CLEAR", "MAX PLAY" 등) 및 씬 판정 알고리즘이 구현되어 있지 않다.

### 2.2. ROI(관심 영역) 및 OCR 좌표의 불일치
* **현황**: 곡명, 난이도, 판정 등의 정보를 읽어오는 ROI 좌표는 프리스타일 선곡 화면에 맞춰져 있다.
* **한계**: 결과 화면은 선곡 화면과 레이아웃(곡 이미지 위치, 판정 텍스트 크기 및 좌표)이 완전히 다르다. 결과 화면 전용의 ROI 프리셋이 추가되지 않으면 정확한 곡 정보와 최종 판정을 읽어낼 수 없다.

### 2.3. 기록 갱신 판정의 선후 관계 꼬임
* **현황**: 현재 Overmax는 프리스타일 화면에서 감지된 정보가 안정화(`is_stable = true`)되었을 때 비로소 DB에 기록을 기록/갱신한다.
* **한계**: 결과 화면에서 캡처하려면 **"결과 화면 감지 -> 이전 최고 기록 대조 -> 신기록 확인 -> 화면 캡처 저장 -> DB 갱신"**의 순서로 정교하게 작동해야 하는데, 현재의 원자적 플레이 컨텍스트(`PlayContext`) 동기화 구조는 인게임 플레이 후 프리스타일 화면으로 완전히 복귀했을 때 작동하도록 치우쳐 있다.

### 2.4. 디스크 I/O 및 이미지 인코딩 병목
* **현황**: 캡처 엔진은 실시간 프레임을 스레드 간에 빠르게 전달해야 한다.
* **한계**: 캡처된 생 버퍼(Raw BGRA) 데이터를 PNG나 JPEG로 인코딩하고 디스크에 쓰는 디바이스 작업은 CPU 및 I/O 연산 집약적이다. 이를 동기식으로 처리하면 화면 분석 루프나 GUI 렌더링 스레드가 일시적으로 멈추는 미세한 병목(Stuttering)이 인게임 플레이 직후 혹은 메뉴 전환 시 발생할 수 있다.

## 3. 우아한 해결을 위한 설계 방향성 (Proposed Architecture)

위 한계점들을 해결하고 깔끔하게 기능을 구현하기 위한 구조적 제안은 다음과 같다.

### 3.1. 비동기 이미지 인코더 파이프라인 (Asynchronous Writer)
* 캡처 프레임을 파일로 저장할 때, 디텍션 스레드나 GUI 스레드에서 직접 인코딩하지 않는다.
* 캡처 직후의 Raw 픽셀 데이터와 저장할 경로명만 `mpsc channel`을 통해 전용 백그라운드 파일 쓰기 워커 스레드로 즉시 넘긴다.
* 메인 파이프라인은 0.1ms 이내로 채널 전송을 마치고 화면 감지를 계속 진행하며, 디스크 I/O 및 PNG 압축 연산은 백그라운드에서 백그라운드 스레드가 저지연으로 전담하여 처리한다.

### 3.2. 상태 전이 다이어그램 기반 상태 제어
* 단순히 씬을 매 프레임 스캔하는 대신, 다음과 같이 상태 머신을 고도화하여 스캔 비용을 줄인다.
  1. `SceneType::Freestyle` (선곡 완료)
  2. `SceneType::InGame` (플레이 중: 오버레이 비활성화, 스캔 최소화)
  3. `SceneType::Result` (결과 화면 진입: 1회 스캔 후 신기록 여부 판단 및 캡처 트리거)
  4. `SceneType::Freestyle` (복귀)

### 3.3. 옵트인 설정 및 해상도 최적화
* 스크린샷 캡처는 드라이브 용량을 지속적으로 소모하므로 설정(`settings.user.json`)에 토글 옵션을 추가한다.
* 캡처 시 불필요한 오버레이 창 영역은 제외하고, 순수 게임 창의 클라이언트 영역만 잘라서 저장한다.
