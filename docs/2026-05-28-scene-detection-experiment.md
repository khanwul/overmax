# 템플릿 매칭 기반 씬 감지(Scene Detection) 실험 보고서

* **작성일**: 2026년 5월 28일
* **실험 브랜치**: `feat/scene-detect-experiment`

---

## 1. 실험 목적
DJMAX RESPECT V 오버레이의 자동 기록 및 추천 시스템은 현재 화면이 "곡 선택 화면(Freestyle/Online)"인지 확인하기 위해 로고 영역을 Windows OCR 엔진으로 주기적으로 분석합니다.
* **문제점**: OCR 엔진은 매우 안정적이지만 연산 비용(CPU/GPU)이 높습니다. 
* **해결 제안**: 선곡 화면이 처음 인식되어 안정화된 후 화면 픽셀의 16×9 템플릿(Feature Vector)을 학습하고, 이후 프레임부터는 단순 L2 Distance(평균 제곱근 오차) 비교 연산만 수행함으로써 OCR 호출 주기를 줄이고 CPU 점유율을 최적화하고자 했습니다.

---

## 2. 시도 및 구현 내용

### A. 16×9 그리드 기반 전체 화면 템플릿 매칭
* 화면 뷰포트를 16×9 그리드로 나누어 144차원 Grayscale 피처 벡터를 추출.
* 씬이 안정 상태(`is_stable`)이고 곡 매칭이 확인되었을 때, 템플릿 벡터를 누적하여 디스크(`cache/scene_templates.bson`)에 캐싱 및 자동 로드.
* 첫 프레임 블렌딩 시 학습 속도 저하(Cold Start)를 막기 위해 최초 학습 시 `alpha = 1.0`로 즉시 초기화하는 로직 추가.

### B. 동적 가중치(Temporal MAD) 알고리즘 도입
* 화면 중앙에서 상시 움직이는 BGA(백그라운드 애니메이션/비디오)의 노이즈를 억제하기 위해 각 그리드 셀의 **시간에 따른 평균 절대 편차(MAD)**를 함께 학습.
* L2 Distance 계산 시 편차가 큰 셀(BGA)에는 낮은 가중치를, 편차가 극도로 작은 정적 영역(테두리, 메뉴 고정 프레임 등)에는 높은 가중치를 배정하는 `compute_weighted_l2_distance_f32_u8` 연산 적용.

### C. 디버그 오버레이 UI 구현
* 오버레이 헤더 영역에 실시간 감지 상태 램프(Teal = 템플릿 작동 중, Yellow = OCR 판정 중, Gray = 미감지/캐싱) 및 툴팁을 추가하여 매칭 상태를 시각적으로 감시할 수 있도록 지원.

## 3. 실패 원인 분석

### 1) UI 구성 요소 변동성에 대한 취약성 (가장 정적인 곳의 역설)
* NM / HD / MX / SC 난이도 버튼이나 곡 목록 프레임은 사용자가 조작하기 전까지 **화면에서 가장 정적이고 변화가 없는 영역**입니다.
* 이로 인해 MAD 알고리즘은 이 영역을 "매우 신뢰할 수 있는 영역"으로 인식해 극도로 높은 가중치(가장 좁은 오차 편차 허용)를 부여합니다.
* 하지만 사용자가 패턴이나 난이도를 바꾸는 즉시, 이 초고가중치 영역의 픽셀 값이 크게 변하게 되며, 이 미세한 변화로 인해 L2 오차가 수천 배 스파이킹하면서 임계값을 초과해 버립니다.
* 결과적으로 **패턴을 변경할 때마다 템플릿 매칭이 풀리고 노란색(OCR) 판정으로 강제 폴백**되는 비효율이 발생하였습니다.

### 2) 반투명 헤더 레이아웃과 BGA 투과 문제
* 전체 화면 대신 로고 영역(상단 왼쪽)만 떼어내 템플릿 혹은 이진화(Binarization) 비교를 시도하려 했으나, DJMAX 헤더 바의 검은 배경판이 **반투명(Semi-transparent)** 처리되어 있어 그 뒤로 BGA 비디오가 뚫고 보입니다.
* 이전에 시도했던 Otsu 이진화 등도 BGA에 백색 연출이나 강한 섬광이 지나갈 때 로고의 흰색 부분과 합쳐지면서 임계값이 뭉개지고 형태가 깨지는 현상이 발생했습니다.
* 결국 BGA 투과 노이즈가 있는 곳에서는 단순한 픽셀 비교/이진화 매칭의 신뢰도가 OCR에 비해 훨씬 낮았습니다.

## 4. 결론 및 향후 대안

BGA의 반투명 투과 특성 및 난이도/패턴 선택 상태에 따른 UI 변동성 때문에, 선곡 화면을 **픽셀 템플릿 매칭 방식으로 안정되게 잡는 것은 한계**가 큽니다. 

따라서 다음 대안을 권장합니다:

### 💡 추천 대안: 스마트 하트비트 OCR + 자켓 기반 고속 퇴출 (Fast Exit)
화면 픽셀 비교 대신 **OCR 엔진의 스케줄링 주기를 스마트하게 조정**하여 CPU 점유율을 줄이는 방식입니다.
1. **변동식 주기 (Variable Cooldown)**:
   * 씬 미감지 상태(`Unknown`) 시: 씬 진입을 빠르게 감지해야 하므로 **0.5초** 주기로 OCR 수행.
   * 씬 감지 완료 상태(`Freestyle/Online`) 시: 이미 씬이 특정되었으므로 OCR 주기를 **3.0초 ~ 5.0초(하트비트)**로 대폭 완화하여 연산 비용을 거의 0에 가깝게 유지.
2. **이벤트 기반 즉각 퇴출 (Fast Exit)**:
   * 5초 주기를 기다리는 도중 씬을 이탈하는 것을 빠르게 감지하기 위해, **자켓 소실**(`validate_jacket_presence`가 1.5초간 실패) 또는 **인게임 감지** 시 즉각 OCR 타이머와 상관없이 `Unknown` 씬 상태로 전환.
3. **효과**: 픽셀 비교의 불안정성이나 예외 상황(BGA 밝기 변화, 테마 변경)을 전혀 타지 않으면서도, OCR 연산 횟수를 기존 대비 90% 이상 차단할 수 있습니다.
